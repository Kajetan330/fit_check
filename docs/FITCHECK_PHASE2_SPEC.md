# FitCheck Phase 2 Implementation Plan

Last updated: July 14, 2026

Scope: trust cleanup, real Supabase auth, booking conversion, and edits-lite. This plan reflects the current codebase after the creator-commerce scaffold and the removal of the Style Quiz.

Every workstream should finish with:

```bash
npm run check
```

Keep changes small and shippable. Do not remove the seeded/local demo flow until the Supabase-backed replacement is implemented and tested.

- Stripe checkout, webhook, checkout status, creator commerce, and entitlement scaffolding exist.
- Supabase migrations exist in `supabase/migrations/`.

## Workstream C: Real Supabase Authentication

Status: not implemented. This starts with dashboard steps the project owner must do.

Goal: replace demo identity with real Supabase auth while preserving lightweight local personalization until later.

### Owner Dashboard Steps

In Supabase:

1. Go to Auth -> Providers -> Google.
2. Enable Google OAuth.
3. Create or connect a Google Cloud OAuth client.
4. Add the Supabase callback URL in Google Cloud:
   - `https://<project-ref>.supabase.co/auth/v1/callback`
5. Go to Auth -> Providers -> Email.
6. Enable magic link or OTP sign-in.
7. Disable password signups for launch simplicity.
8. Go to Auth -> URL Configuration.
9. Set Site URL:
   - `https://fit-check-ecru.vercel.app`
10. Add redirect URLs:
   - `https://fit-check-ecru.vercel.app/**`
   - `http://localhost:5173/**`
   - your custom domain when added.

### New Database Migration

Add a profiles table backed by `auth.users`:

```sql
create table public.profiles (
  id uuid primary key references auth.users(id) on delete cascade,
  display_name text,
  role text not null default 'customer'
    check (role in ('customer', 'creator', 'admin')),
  created_at timestamptz not null default now()
);

alter table public.profiles enable row level security;

create policy "read own profile" on public.profiles
  for select using (auth.uid() = id);

create policy "update own profile" on public.profiles
  for update using (auth.uid() = id)
  with check (auth.uid() = id);

create function public.handle_new_user()
returns trigger
language plpgsql
security definer
set search_path = public
as $$
begin
  insert into public.profiles (id, display_name)
  values (
    new.id,
    coalesce(new.raw_user_meta_data->>'full_name', split_part(new.email, '@', 1))
  );
  return new;
end;
$$;

create trigger on_auth_user_created
  after insert on auth.users
  for each row execute function public.handle_new_user();
```

Role escalation must happen only through service-role/admin actions, never from client UI.

### Client Changes

- Replace local `signIn(name, role, email)` identity with Supabase auth.
- Add:
  - `signInWithGoogle`
  - `signInWithEmail`
  - `signOut`
- Listen to `supabase.auth.onAuthStateChange`.
- On sign-in, fetch the user's `profiles` row and populate `state.user`.
- Stop persisting `state.user` into FitCheck localStorage. Supabase should own the session.
- Keep localStorage for non-critical local MVP state for now:
  - saved creators
  - saved posts
  - closet items
  - local studio demo drafts
- Add `/auth/callback` to finish OAuth/magic-link redirects.

### Booking Draft Preservation

When a logged-out user starts booking:

- store the intended return path in `sessionStorage["fitcheck-auth-redirect"]`;
- store draft booking answers/uploads in `sessionStorage["fitcheck-draft-booking-v1"]`;
- after `/auth/callback`, return to the booking draft instead of restarting.

Acceptance criteria:

- Google sign-in works.
- Magic-link sign-in works.
- A profile row is created for new users.
- Session survives reload.
- No UI lets a user select or escalate role.
- Authenticated checkout records can be associated to the real user.
- RLS prevents reading another user's private profile data.
- `npm run check` passes.

## Workstream D: Conversion And Booking Wizard

Status: not implemented.

Goal: make booking feel like a real mobile checkout flow instead of a long prototype form.

### D1 Booking Wizard

Replace the current long intake with a five-step wizard:

1. `service`
   - Show service outcome, price, turnaround, deliverables, and scope.
2. `brief`
   - Render the selected service's intake prompts.
   - First required prompts visible; optional detail collapsed.
3. `uploads`
   - Use existing upload helpers.
   - Support mobile camera uploads with `accept="image/*"` and `capture="environment"`.
   - Show thumbnails and remove actions.
4. `review`
   - Show answers, uploads, price, turnaround, and payment-hold explanation.
   - If logged out, show auth inline and preserve draft.
5. `pay`
   - Start trusted checkout.

Use a component-local reducer and persist the draft to `sessionStorage["fitcheck-draft-booking-v1"]`.

Mobile should have a sticky bottom CTA with price and next action.

### D2 Service Presentation

Extend service presentation with optional fields:

```ts
exampleResultImage?: string;

```

Populate all seeded services and render:

- outcome;
- price;
- turnaround;
- deliverables;
- who it is for;
